## Problem Statement

Deft Directive is a Markdown framework for AI agents to use when generating software. It defines layered behavioral rules, workflow strategies, and quality gates — but it does not have a specification for itself. This creates a compounding problem: agents working on deft fill gaps with guesses, bugs accumulate without authoritative acceptance criteria, and contributors have no clear picture of what deft is supposed to do. Twelve open Phase 1 issues are actively blocking user adoption.

---

## Goals

1. **Close the dogfooding gap** — deft must have a SPECIFICATION.md that defines its own behavior (closes #67)
2. **Fix adoption blockers** — resolve all Phase 1 bugs preventing reliable first-run experience
3. **Clean the content layer** — remove stale references, deduplication debt, and leaked personal data from the framework files
4. **Establish CI** — automated lint + test on every PR; no more unguarded merges

## User Stories

**US-1** — As a new user, I want deft-setup to correctly identify my project name and tech stack so that my PROJECT.md reflects my actual project, not the deft framework internals.
*(Closes #79, #80)*

## Functional Requirements

### Phase 1 — Bug Fixes & Adoption Blockers

- **FR-1**: deft-setup Phase 2 MUST NOT scan `./deft/` for build files (`go.mod`, `package.json`, `pyproject.toml`, `Cargo.toml`, `*.csproj`) when inferring project identity. *(#79)*
- **FR-2**: deft-setup Phase 2 MUST NOT run `git` commands inside `./deft/` to determine project identity. *(#79)*
- **FR-3**: deft-setup Phase 2 MUST provide a fallback project name prompt when no build files exist at the project root. *(#80)*
- **FR-4**: USER.md template and Phase 1 interview MUST NOT include a Primary Languages field. Language is a project-level concern resolved in Phase 2 via codebase inference. *(#107)*
- **FR-5**: deft-setup Phase 2 MUST ask about deployment platform before language, using the platform to narrow the language shortlist. *(#108)*
- **FR-6**: All vBRIEF files generated by deft (via `run spec` or agent skill) MUST conform to the vBRIEF schema at `vbrief.org`. Specifically: top-level key MUST be `plan` (not `title`) for the spec name; task status values MUST be from `pending`/`running`/`completed`/`blocked`/`cancelled` (not legacy `todo`/`doing`/`done`); top-level `status` MUST be `approved` before rendering. *(#72)*
- **FR-7**: `run bootstrap` MUST NOT enter an infinite loop when `strategies/` is empty or unresolvable. *(#91, #92)*
- **FR-8**: `strategies/default.md` MUST be merged into `strategies/interview.md` and removed. *(#31)*
- **FR-9**: `strategies/brownfield.md` MUST be updated to redirect to `strategies/map.md`; standalone redundant content removed. *(#50)*
- **FR-10**: All `run` CLI commands MUST display the version on startup. *(#49)*
- **FR-11**: `coding/` MUST include a build output validation directive for custom build scripts, covering silent `dist/` failures. *(#105)*
- **FR-12**: A toolchain/environment validation gate MUST be documented in the framework — agents MUST verify required toolchain is installed before beginning implementation. *(#106)*

### Phase 2 — Content Fixes

- **FR-13**: All references to legacy `USER.md` and `PROJECT.md` paths MUST resolve to their canonical locations (`~/.config/deft/USER.md`, `./PROJECT.md`). *(#58)*
- **FR-14**: `history/changes/` directory MUST exist with a `README.md` documenting the change lifecycle structure. *(#59)*
- **FR-15**: `strategies/yolo.md` MUST reference `interview.md` for shared phases (sizing gate, chaining gate, acceptance gate) rather than duplicating them. *(#23)*
- **FR-16**: `strategies/speckit.md` MUST include a `**⚠️ See also**` cross-reference banner. *(#24)*
- **FR-17**: `commands.md` vBRIEF example MUST use status vocabulary consistent with `vbrief/vbrief.md`. *(#25)*
- **FR-18**: `PROJECT.md` (legacy location) MUST NOT contain private project configuration (Voxio Bot content); replace with a generic template or remove.
- **FR-19**: `contracts/hierarchy.md` MUST be created documenting the dual-hierarchy framework (durability axis: Standards > APIs > Specs > Code; generative axis: Spec → Contracts → Code). *(#84 Phase 1)*
- **FR-20**: `main.md` MUST include adaptive teaching behavior: concise when recommendations are accepted without question; explanatory when recommendations are questioned or overridden. *(#84 Phase 1)*
- **FR-21**: `strategies/interview.md` MUST include a `! State WHY` rule — when making an opinionated recommendation, state the principle in one sentence. *(#84 Phase 1)*
- **FR-22**: `CONTRIBUTING.md` MUST document: dev environment setup, how to run tests (`task test`, `task check`), how to run the CLI locally (`python run`), and how to build the Go installer (`go build ./cmd/deft-install/`). *(#67 AC item 3)*
- **FR-23 (depends on #89)**: README.md MUST be reframed — remove "Warping Process", "What is Warping?", "Contributing to Warping" references; update tagline per #89 resolution.
- **FR-24 (depends on #89)**: `meta/philosophy.md` MUST be created per #84 Phase 2 framing, once #89 is resolved.

### Phase 3 — CI

- **FR-25**: A GitHub Actions workflow MUST run on every PR and push to master: Python lint (`ruff`), type check (`mypy`), test suite (`pytest`), and Go build check (`go build ./cmd/deft-install/`). *(#57)*
- **FR-26**: The CI workflow MUST use action versions that support the GitHub Actions Node.js 24 runtime. Recommended versions (consistent with existing `release.yml`): `actions/checkout@v4`, `actions/setup-python@v5`, `actions/setup-go@v5`, `actions/upload-artifact@v4`, `actions/download-artifact@v4`.
- **FR-27**: CI MUST fail if test coverage falls below 85% on measurable code (excludes `run` and `run.py` — see NFR-2).

## Non-Functional Requirements

- **NFR-1**: Test coverage MUST be ≥85% on all measurable Python code. *(Resolves stated-vs-enforced gap)*
- **NFR-2**: `run` and `run.py` are explicitly excluded from coverage measurement. This exclusion MUST be documented in `pyproject.toml` with a comment explaining the rationale (terminal-only path; pending dedicated tracking issue for future refactor). A new GitHub issue MUST be opened to track bringing `run` into coverage.
- **NFR-3**: Framework MUST support Windows, macOS (Intel + Apple Silicon), and Linux (amd64 + arm64) — no platform regressions.
- **NFR-4**: All vBRIEF files generated by any deft entry point (CLI, agent skill) MUST pass `task spec:validate` without manual correction.
- **NFR-5**: `task check` MUST serve as the authoritative pre-commit gate — passing `task check` is the definition of "ready to commit" for all code changes.
